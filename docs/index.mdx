<h2>The problem</h2>

<p>When your application needs to make outbound HTTP requests at scale — scraping, API polling, data collection — you run into rate limits, IP blocks, and connection failures. Managing a pool of proxy IPs manually means writing retry logic, tracking which IPs are healthy, rotating through them fairly, and handling failures gracefully. Every project reinvents this wheel.</p>

<p>Proxy Hopper solves it once, as a standalone service.</p>

<img src="/images/diagram-light.svg" alt="Proxy Hopper architecture" className="overview-diagram block dark:hidden"/>
<img src="/images/diagram-dark.svg" alt="Proxy Hopper architecture" className="overview-diagram hidden dark:block"/>

<p className="mb-xl">Point your HTTP client at Proxy Hopper as its proxy. Proxy Hopper handles everything else.</p>

<hr/>

<h2 className="mb-lg">How it works</h2>

<Steps>
  <Step title="Configure your proxy providers and targets">
    Define your external proxy IPs — grouped by provider with credentials and region tags. Set targets as URL regex patterns, each with its own pool, rate limit, and retry policy.
  </Step>
  <Step title="Point your HTTP client at Proxy Hopper">
    Use standard HTTP proxy settings, HTTPS CONNECT tunnels, or forwarding mode. No changes to your application logic.
  </Step>
  <Step title="Proxy Hopper handles the rest">
    IP rotation, rate limiting between requests, automatic quarantine of failing IPs, retries on failure, and health probing in the background.
  </Step>
</Steps>

<hr/>

<h2 className="mb-lg">Key features</h2>

<CardGroup cols={3}>
  <Card title="IP rotation" icon="rotate">
    Each target maintains its own FIFO queue. IPs are held off the pool between requests to respect rate limits.
  </Card>
  <Card title="Automatic quarantine" icon="shield-halved">
    IPs that accumulate failures are quarantined automatically and released back into rotation after a configurable timeout.
  </Card>
  <Card title="Per-target policies" icon="sliders">
    Different targets can have different rate limits, retry counts, and quarantine thresholds — match the policy to the site.
  </Card>
  <Card title="Three integration modes" icon="plug">
    HTTP proxy, HTTPS CONNECT tunnel, and URL-forwarding — all using the same rotation and retry logic.
  </Card>
  <Card title="Redis HA" icon="server">
    In-memory backend for single-instance deployments. Redis backend for multi-instance HA with shared pool state.
  </Card>
  <Card title="Prometheus metrics" icon="chart-line">
    Full observability — request rates, retry counts, queue depth, quarantine events, and per-IP probe status.
  </Card>
</CardGroup>

<h2>Three integration modes</h2>

<p>Proxy Hopper supports three ways for clients to send requests. All three use the same IP rotation and retry logic.</p>

<table className="overview-table">
  <thead>
    <tr>
      <th>Mode</th>
      <th>How to use</th>
      <th>HTTPS retries</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td><strong>HTTP proxy</strong></td>
      <td><code>http_proxy=http://proxy-hopper:8080</code></td>
      <td>Yes</td>
    </tr>
    <tr>
      <td><strong>CONNECT tunnel</strong></td>
      <td><code>https_proxy=http://proxy-hopper:8080</code></td>
      <td>No — tunnel is opaque</td>
    </tr>
    <tr>
      <td><strong>URL forwarding</strong></td>
      <td><code>X-Proxy-Hopper-Target</code> header</td>
      <td>Yes — recommended for APIs</td>
    </tr>
  </tbody>
</table>

<Tip>
  Use forwarding mode for HTTPS APIs where you need retry support. CONNECT tunnels are opaque byte relays — Proxy Hopper cannot retry a mid-flight failure once the TLS handshake is complete.
</Tip>

<h2 className="mb-lg">Get started</h2>

```yaml
# config.yaml
targets:
  - name: general
    regex: '.*'
    ipList:
      - "10.0.0.1:3128"
      - "10.0.0.2:3128"
    minRequestInterval: 1s
    numRetries: 3
```

```bash
docker run -v $(pwd)/config.yaml:/etc/proxy-hopper/config.yaml \
  ghcr.io/cams-data/proxy-hopper:latest
```

```python
import requests
resp = requests.get(
    "https://example.com",
    proxies={"https": "http://localhost:8080"}
)
```
